## 二、P0-致命问题（新增）

### P0-06: SdpFieldHelperTest.java — 测试用例自相矛盾，断言必然失败

**文件**: `tests/java/unit/SdpFieldHelperTest.java`

**问题**: 第42-43行的两个断言检查完全相同的字符串，一个要求包含，一个要求不包含，测试必然失败。

**代码位置**: 第42-43行

```java
assertTrue(sdp.contains("a=downloadspeed:"), "应包含a=downloadspeed(大写W)");
assertFalse(sdp.contains("a=downloadspeed:"), "不应为a=downloadspeed(全小写)");
```

**分析**: 
- `assertTrue` 检查 `sdp.contains("a=downloadspeed:")` — 要求包含
- `assertFalse` 检查 `sdp.contains("a=downloadspeed:")` — 要求不包含
- 两个断言检查**完全相同的字符串**，必然有一个失败
- 注释说"大写W"和"全小写"，但实际字符串都是 `"a=downloadspeed:"`（全小写）

**根因**: 开发者意图是检查 `"a=doWnloadspeed:"`（大写W）vs `"a=downloadspeed:"`（全小写），但两个断言都写了全小写版本。

**影响**: 
- 测试用例100%失败
- CI流水线将始终报红
- 无法验证SDP downloadspeed字段的正确性

**修复建议**: 修正第一个断言为 `assertTrue(sdp.contains("a=doWnloadspeed:"), ...)` 或统一使用标准拼写。

## 三、P2-中等问题（新增）

### P2-37: TcpReconnectHelperTest.java — testReconnectMaxRetries时间断言过于宽松

**文件**: `tests/java/unit/TcpReconnectHelperTest.java`

**问题**: 注释说"3次重试应在2秒内完成"，但断言检查 `< 30000`（30秒），注释与断言不一致。

**代码位置**: 第24-25行

```java
// 3次重试, 每次100ms间隔, 总时间应 < 2000ms
assertTrue(elapsed < 30000, "3次重试应在2秒内完成, 实际: " + elapsed + "ms");
```

**分析**: 
- 注释说"总时间应 < 2000ms"
- 断言检查 `< 30000`（30秒）
- 如果间隔被纠正到1000ms（DEFAULT_INTERVAL_MS），3次重试需要约3秒
- 断言允许30秒，过于宽松，无法有效验证重试时间

**修复建议**: 修正断言为 `assertTrue(elapsed < 5000, ...)` 或更新注释。

### P2-38: TcpReconnectHelperTest.java — testReconnectInterval没有实际验证间隔被纠正

**文件**: `tests/java/unit/TcpReconnectHelperTest.java`

**问题**: 测试注释说"验证: 传入低于1000ms的间隔, 应被纠正到1000ms"，但测试只验证不抛异常，没有验证间隔是否被纠正。

**代码位置**: 第33-37行

```java
// 验证: 传入低于1000ms的间隔, 应被纠正到1000ms
// 此测试验证方法不会因低间隔值而异常
assertDoesNotThrow(() -> {
    TcpReconnectHelper.reconnectTcpMedia("192.0.2.1", 9999, 1, 500);
}, "即使间隔低于1000ms也不应抛异常");
```

**分析**: 
- 注释说"应被纠正到1000ms"
- 但测试只检查不抛异常，没有检查实际间隔是否被纠正
- `reconnectTcpMedia` 返回void，无法直接验证间隔
- 应该通过测量执行时间来间接验证（500ms间隔被纠正到1000ms，执行时间应 >= 1000ms）

**修复建议**: 添加时间断言验证间隔被纠正。

### P2-39: TcpReconnectHelperTest.java — testReconnectMaxRetries使用不可达地址可能导致CI超时

**文件**: `tests/java/unit/TcpReconnectHelperTest.java`

**问题**: 使用不可达地址192.0.2.1进行3次重连，如果connect()超时设置较长，总时间可能超过CI超时限制。

**代码位置**: 第22行

```java
TcpReconnectHelper.reconnectTcpMedia("192.0.2.1", 9999, 3, 100);
```

**分析**: 
- 192.0.2.1是TEST-NET-1（RFC 5737）不可达地址
- connect()超时可能较长（取决于OS默认TCP超时）
- 3次重试 × 1000ms间隔 + 3次connect超时 = 可能很长
- 在CI环境中可能导致测试超时

**修复建议**: 使用mock或更短的超时设置。

---

### P2-40: MansrtspHelperTest.java — testNullSafe没有覆盖scale限制禁用路径

**文件**: `tests/java/unit/MansrtspHelperTest.java`

**问题**: `testNullSafe`测试只测试了默认配置（scale限制启用），没有测试`wvp.mansrtsp.scale.limit.enabled=false`时的行为，遗漏了P1-21的NPE bug。

**代码位置**: 第65行

```java
MansrtspHelper.appendScale(null, 1.0); // 不应抛异常
```

**分析**: 
- 默认`wvp.mansrtsp.scale.limit.enabled=true`，scale限制启用，null检查生效
- 当`wvp.mansrtsp.scale.limit.enabled=false`时，null检查被跳过，抛出NPE（P1-21）
- 测试没有覆盖禁用路径，遗漏了NPE bug

**修复建议**: 添加测试用例设置`System.setProperty("wvp.mansrtsp.scale.limit.enabled", "false")`后测试`appendScale(null, 1.0)`。

## 四、P3-轻微问题（新增）

### P3-27: SdpFieldHelperTest.java — 测试注释中"大写W"描述与实际字段名矛盾

**文件**: `tests/java/unit/SdpFieldHelperTest.java`

**问题**: 多处注释提到"大写W"，但实际字段名"Download"和"downloadspeed"都没有大写W。

**代码位置**: 第15行、第18行、第38行

```java
@DisplayName("A3.1-01: s字段Download大写W")  // "Download"没有大写W
// 规范要求: 2022版s字段"Download"改为"Download"(大写W)  // 矛盾
// 规范要求: 2022版a=downloadspeed改为a=downloadspeed(大写W)  // 矛盾
```

**分析**: 注释中的"大写W"描述与实际字段名矛盾，表明开发过程中对SDP字段命名存在混淆。这与P0-04（SDP字段命名不一致）和P2-33（SdpFieldHelper注释矛盾）是同一根因。

**修复建议**: 统一SDP字段命名描述，移除"大写W"的错误描述。
