## 二、P2-中等问题（新增）

### P2-52: ApiDeviceControlController.java — 固件上传未检查Content-Type

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/controller/ApiDeviceControlController.java`

**问题**: 固件上传只检查文件扩展名，未检查Content-Type。

**代码位置**: 第373行
```java
if (filename == null || !filename.toLowerCase(java.util.Locale.ENGLISH).matches(".*\\.(bin|img|zip)$")) {
```

**分析**: 
- 仅通过文件扩展名判断文件类型
- 攻击者可上传恶意文件并重命名为.bin/.img/.zip
- 未校验`file.getContentType()`

**影响**: 
- 恶意文件可能被上传并存储
- 如果固件文件被直接提供给设备下载，可能导致设备被植入恶意固件

**修复建议**: 增加Content-Type白名单校验：
```java
String contentType = file.getContentType();
if (contentType == null || !ALLOWED_CONTENT_TYPES.contains(contentType)) {
    return WVPResult.fail(400, "不支持的文件类型");
}
```

### P2-53: ApiDeviceControlController.java — 固件上传未检查文件内容魔数

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/controller/ApiDeviceControlController.java`

**问题**: 固件上传未检查文件内容的魔数(magic bytes)，仅依赖扩展名校验。

**分析**: 
- 文件扩展名可被伪造
- 真正的文件类型应通过文件头部魔数判断
- 例如：ZIP文件的魔数是`PK\x03\x04`，BIN/IMG文件通常有特定的固件头部

**影响**: 
- 攻击者可上传伪装成固件的恶意文件
- 可能导致设备刷入恶意固件

**修复建议**: 读取文件前几个字节，校验魔数：
```java
byte[] header = new byte[4];
try (InputStream is = file.getInputStream()) {
    int read = is.read(header);
    // 校验魔数
}
```

### P2-54: ApiDeviceControlController.java — device_upgrade端点SSRF防护不完整

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/controller/ApiDeviceControlController.java`

**问题**: `device_upgrade`端点的SSRF防护只检查URL协议(scheme)，不检查主机地址。

**代码位置**: 第444-454行
```java
if (fileUrl != null && !fileUrl.isEmpty()) {
    try {
        java.net.URI uri = new java.net.URI(fileUrl);
        String scheme = uri.getScheme();
        if (scheme == null || (!"http".equals(scheme) && !"https".equals(scheme))) {
            return WVPResult.fail(400, "fileUrl仅支持http/https协议");
        }
        // 缺少: 主机地址校验，未阻止内网地址
    } catch (java.net.URISyntaxException ex) {
        return WVPResult.fail(400, "fileUrl格式无效");
    }
}
```

**分析**: 
- 只校验协议为http/https
- 未校验主机是否为内网地址（127.0.0.1, 10.x, 172.16-31.x, 192.168.x等）
- 未校验主机是否为云元数据服务地址（169.254.169.254）
- 与P1-06是同一类问题，但影响不同的端点

**影响**: 
- 攻击者可传入`http://127.0.0.1:8080/admin`等内网地址
- 设备会从内网地址下载"固件"，可能导致内网信息泄露
- 设备可能被引导下载恶意固件

**修复建议**: 增加主机地址校验，阻止内网地址和云元数据地址。

### P2-55: ApiDeviceControlController.java — device_upgrade方法缺少并发升级检查

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/controller/ApiDeviceControlController.java`

**问题**: `device_upgrade`方法缺少并发升级检查，注释明确指出需要但未实现。

**代码位置**: 第441-442行
```java
// 设备升级前检查: 生产环境应查询设备当前升级状态，防止并发升级冲突
// 建议: deviceService.isDeviceUpgrading(deviceId) → 返回"设备正在升级中"
```

**分析**: 
- 注释明确指出需要检查设备是否正在升级
- 但实际代码未实现此检查
- 多个并发升级请求可能同时发送给同一设备

**影响**: 
- 并发升级可能导致设备状态混乱
- 设备可能刷入不完整的固件，导致变砖
- 生产环境中是严重风险

**修复建议**: 实现并发升级检查：
```java
if (deviceService.isDeviceUpgrading(deviceId)) {
    return WVPResult.fail(409, "设备正在升级中，请稍后重试");
}
```
