## 二、P0-致命问题（新增）

### P0-04: SDP字段命名不一致 — 代码与测试期望值矛盾

**文件**: 
- 代码: `news/main/java/com/genersoft/iot/vmp/gb28181/utils/SdpFieldHelper.java`
- 测试: `tests/sipp/sdp/sdp_download_sfield.xml`, `tests/sipp/sdp/sdp_downloadspeed.xml`

**问题**: 代码生成的SDP字段名与sipp测试期望的字段名不一致，导致测试必然失败。

**代码实际使用**:
```java
// SdpFieldHelper.java 第103行
sdpBuilder.append("s=Download\r\n");  // 标准拼写

// SdpFieldHelper.java 第167行
sdpBuilder.append("a=downloadspeed:").append(speed).append("\r\n");  // 全小写
```

**测试期望值**:
```xml
<!-- sdp_download_sfield.xml -->
<ereg regexp="s=DoWnload">  <!-- 大写W -->

<!-- sdp_downloadspeed.xml -->
<ereg regexp="a=doWnloadspeed:4">  <!-- 大写W -->
```

**影响**: 
- sipp测试将100%失败，因为代码生成"Download"但测试期望"DoWnload"
- 无法验证SDP字段是否正确
- 补丁文档`04-SIPCommander.md`声称"DoWnload"是"规范特殊拼写"，但实际GB28181-2022标准使用标准英文"Download"

**修复建议**: 统一使用标准英文拼写"Download"和"downloadspeed"，修正测试和补丁文档中的"DoWnload"/"doWnloadspeed"。

## 三、P1-严重问题（新增）

### P1-13: XML元素命名不一致 — 命令发送与结果解析使用不同字段名

**文件**:
- 命令发送: `backend/.../SIPCommander2022Supplement.java`
- 结果解析: `news/.../DeviceUpgradeResultNotifyMessageHandler.java`

**问题**: 发送设备升级命令和解析升级结果通知时，使用不同的XML元素命名。

**命令发送方** (SIPCommander2022Supplement.java):
```java
// 第249行: FirmWare (W大写)
xmlBuilder.append("<FirmWare>").append(firmwareUrl).append("</FirmWare>");
// 第250行: FileuRL (u小写)
xmlBuilder.append("<FileuRL>").append(fileSize).append("</FileuRL>");
// 第252行: sessionID (s小写)
xmlBuilder.append("<sessionID>").append(sessionId).append("</sessionID>");
```

**结果解析方** (DeviceUpgradeResultNotifyMessageHandler.java):
```java
// 第176行: Firmware (标准拼写)
String firmware = XmlUtil.getText(rootElement, "Firmware");
// 第177行: SessionID (S大写)
String sessionId = XmlUtil.getText(rootElement, "SessionID");
```

**影响**: 如果设备回显升级命令中的字段名（FirmWare/sessionID），结果处理器将无法解析（因为它查找Firmware/SessionID），导致升级结果丢失。

**修复建议**: 统一XML元素命名，建议使用标准拼写"Firmware"和"SessionID"。

### P1-14: sipp测试reg_sm3_auth.xml — SM3响应变量未填充，测试无效

**文件**: `tests/sipp/register/reg_sm3_auth.xml`

**问题**: 测试声明了`[$sm3_response]`和`[$sm3_nonce]`全局变量，但从未填充值。测试注释说"需外部SM3计算引擎填充"，但没有提供任何机制来填充这些变量。

**代码位置**: 第27行声明，第48行引用

```xml
<!-- 第27行: 声明全局变量 -->
<Reference variables="[$sm3_response];[$sm3_nonce]" />

<!-- 第48行: 引用未填充的变量 -->
Authorization: Digest response="[$sm3_response]"
```

**影响**: 
- SIPp发送的REGISTER请求中，response字段为空或未定义
- 服务器将拒绝认证（401/403），测试无法验证SM3认证流程
- 测试实际上无法测试任何东西

**修复建议**: 
1. 使用SIPp的`<exec>`元素配合外部脚本计算SM3摘要
2. 或使用预计算的SM3响应值（固定测试向量）

### P1-15: sipp测试reg_redirect_302.xml — 混合UAS/UAC模式，结构无效

**文件**: `tests/sipp/register/reg_redirect_302.xml`

**问题**: 单个SIPp场景文件混合了UAS模式（接收REGISTER）和UAC模式（发送REGISTER到新地址），但SIPp场景不能同时作为UAS和UAC。

**代码位置**: 第15-40行

```xml
<!-- UAS模式: 接收REGISTER -->
<recv request="REGISTER" />

<!-- UAS模式: 发送302响应 -->
<send>
  SIP/2.0 302 Moved Temporarily
  ...
</send>

<!-- UAC模式: 发送新REGISTER到重定向地址 -->
<send>
  REGISTER sip:[$redirect_target] SIP/2.0
  ...
</send>
```

**影响**: SIPp无法在单个场景中同时扮演UAS和UAC角色。测试注释说"需两个SIPp实例"，但场景文件本身的结构不支持这种用法。

**修复建议**: 拆分为两个独立的SIPp场景文件：一个UAS场景（接收REGISTER，返回302）和一个UAC场景（向重定向地址发送REGISTER）。

### P1-16: QueryResultDialog.vue — AbortController创建但signal未传递，竞态防护失效

**文件**: `ui/components/QueryResultDialog.vue`

**问题**: 组件创建了`AbortController`用于取消请求，但`signal`从未传递给axios请求，导致竞态防护完全失效。

**代码位置**: 第222-242行

```javascript
queryCurrentTab() {
  // 创建AbortController
  if (this.abortController) {
    this.abortController.abort()
  }
  this.abortController = new AbortController()
  const signal = this.abortController.signal  // signal被提取
  
  switch (this.activeTab) {
    case 'homePosition':
      this.queryHomePositionData(signal)  // signal传递给方法
      break
    // ...
  }
}

async queryHomePositionData(signal) {
  // ...
  const { data } = await queryHomePosition(this.deviceId, this.channelId)
  // ↑ signal未传递给queryHomePosition！
  if (signal && signal.aborted) return  // 只检查是否aborted，但请求无法被abort
  // ...
}
```

**影响**: 
- 快速切换标签页时，旧请求不会被取消
- 旧请求的响应可能覆盖新请求的数据
- `AbortController`形同虚设，竞态条件仍然存在

**修复建议**: 将`signal`传递给axios请求：`queryHomePosition(this.deviceId, this.channelId, { signal })`。
