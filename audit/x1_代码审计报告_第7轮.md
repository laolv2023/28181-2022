## 二、P2-中等问题（新增）

### P2-45: 补丁文档01 — Device.isVersion2022()使用startsWith("2.")与GBProtocolVersionHelper不一致

**文件**: `patches/01-RegisterRequestProcessor.md`

**问题**: 补丁文档中`Device.isVersion2022()`使用`startsWith("2.")`进行版本匹配，但`GBProtocolVersionHelper.isVersion2022()`使用`equals("2.0")`严格匹配。

**补丁文档代码** (第124-126行):
```java
public boolean isVersion2022() {
    return protocolVersion != null && protocolVersion.startsWith("2.");
}
```

**实际代码** (GBProtocolVersionHelper.java 第80行):
```java
return version.trim().equals("2.0");  // 严格匹配
```

**影响**: 
- 如果设备发送"2.1"，补丁的`isVersion2022()`返回true，但Helper返回false
- 版本检测结果取决于使用哪个方法，导致不一致
- 可能对"2.1"设备错误启用2022版功能（如SM3），但设备可能不支持

**修复建议**: 统一版本检测逻辑，建议使用`equals("2.0")`严格匹配。

---

### P2-46: 补丁文档02 — calculateDigest使用getBytes()无字符集参数

**文件**: `patches/02-DigestServerAuthenticationHelper.md`

**问题**: `calculateDigest`方法中`data.getBytes()`未指定字符集，使用系统默认字符集。

**代码位置**: 第58行
```java
byte[] mdbytes = messageDigest.digest(data.getBytes());  // 无字符集参数
```

**影响**: 
- 不同环境的默认字符集可能不同（UTF-8 vs GBK vs ISO-8859-1）
- MD5路径和SM3路径（`SmUtil.sm3(data)`内部使用UTF-8）的字符集处理不一致
- 可能导致摘要计算结果不同，认证失败

**修复建议**: 使用`data.getBytes(StandardCharsets.UTF_8)`明确指定字符集。

---

### P2-47: 补丁文档02 — DEFAULT_ALGORITHM是可变静态字段

**文件**: `patches/02-DigestServerAuthenticationHelper.md`

**问题**: `DEFAULT_ALGORITHM`声明为`public static String`，是可变静态字段。

**代码位置**: 第31行
```java
public static String DEFAULT_ALGORITHM = DEFAULT_ALGORITHM_2022;  // 可变
```

**影响**: 
- 任何代码都可以修改此字段，影响所有线程
- 在多线程环境下存在竞态条件
- 难以追踪何时被修改

**修复建议**: 
- 改为`private static volatile String`并通过getter/setter访问
- 或使用`AtomicReference<String>`保证线程安全
- 或在应用启动时从配置读取一次，设为`final`

---

### P2-48: 补丁文档02 — selectAlgorithm依赖isVersion2022()的宽松匹配

**文件**: `patches/02-DigestServerAuthenticationHelper.md`

**问题**: `selectAlgorithm`方法依赖`device.isVersion2022()`，而该方法使用`startsWith("2.")`宽松匹配。

**代码位置**: 第71-76行
```java
private String selectAlgorithm(Device device) {
    if (device != null && device.isVersion2022()) {  // startsWith("2.")
        return DEFAULT_ALGORITHM_2022;  // SM3
    }
    return DEFAULT_ALGORITHM_2016;  // MD5
}
```

**影响**: 
- 与P2-45相关：如果设备发送"2.1"，`isVersion2022()`返回true
- 将对"2.1"设备使用SM3算法，但"2.1"设备可能不支持SM3
- 可能导致认证失败

**修复建议**: 使用`GBProtocolVersionHelper.isVersion2022()`进行严格版本检测。

## 三、P3-轻微问题（新增）

### P3-30: 补丁文档01 — extractProtocolVersion使用replaceAll处理头域值

**文件**: `patches/01-RegisterRequestProcessor.md`

**问题**: 第86-87行使用`replaceAll`处理头域值，但`replaceAll`的第一个参数是正则表达式，`HEADER_X_GB_VER + ":"`中的`:`在正则中没有特殊含义，但使用`replace`（字面量替换）更合适。

```java
String version = xGbVerHeader.toString()
        .replaceAll(HEADER_X_GB_VER + ":", "").trim();
```

**修复建议**: 使用`replace`替代`replaceAll`，避免正则解析开销和潜在的特殊字符问题。

### 4.2 关键发现

1. **版本检测逻辑不一致（P2-45, P2-48）**: 补丁文档使用`startsWith("2.")`宽松匹配，实际代码使用`equals("2.0")`严格匹配。这可能导致对"2.1"等版本号的设备行为不一致。

2. **字符集处理不一致（P2-46）**: MD5路径使用`getBytes()`（系统默认字符集），SM3路径使用UTF-8，可能导致摘要计算结果不同。

3. **可变静态字段（P2-47）**: `DEFAULT_ALGORITHM`是可变静态字段，存在线程安全风险。

4. **正则表达式误用（P3-30）**: `replaceAll`用于字面量替换，应使用`replace`。
