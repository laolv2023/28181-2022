## 二、P2-中等问题（新增）

### P2-49: run_all.sh — 缺少set -e，中间命令失败不会被捕获

**文件**: `tests/scripts/run_all.sh`

**问题**: 第4行使用`set -uo pipefail`但没有`set -e`，中间命令失败不会被捕获。

**代码位置**: 第4行
```bash
set -uo pipefail  # 缺少 -e
```

**分析**: 
- `-u`: 未定义变量报错
- `-o pipefail`: 管道失败传播
- 缺少`-e`: 命令失败不退出
- `trap '...' ERR` 只打印消息，不退出（ERR trap在没有`set -e`时不会自动退出）
- 测试脚本调用使用`|| OVERALL_FAIL=1`捕获失败，但中间的`echo`等命令失败不会被捕获

**影响**: 
- 中间命令失败时脚本继续执行
- 可能导致误导性的测试结果

**修复建议**: 使用`set -euo pipefail`。

---

### P2-50: run_sipp_tests.sh — sipp使用-m 1只发送1条消息，不足以完成完整注册流程

**文件**: `tests/scripts/run_sipp_tests.sh`

**问题**: 第22行sipp命令使用`-m 1`限制只发送1条消息，对于需要多轮交互的注册场景（REGISTER -> 401 -> REGISTER -> 200 OK）不足以完成完整流程。

**代码位置**: 第22行
```bash
sipp "$WVP_ADDR" -sf "$scenario" -m 1 -timeout 10s -timeout_error 2>&1 | tee -a "$SCRIPT_DIR/sipp_stderr.log"
```

**分析**: 
- `-m 1`: 限制SIPp只处理1次呼叫（1次完整场景）
- 对于UAC模式的注册场景，1次呼叫 = 1次完整注册流程（REGISTER -> 401 -> REGISTER -> 200 OK）
- 但对于需要多次交互的场景，`-m 1`可能导致场景提前终止
- 实际上`-m 1`对于单次注册场景是合适的，但对于多步骤场景可能不够

**影响**: 
- 部分测试场景可能无法完成完整交互
- 测试结果可能不准确

**修复建议**: 根据场景类型调整`-m`参数，或移除限制。

### P2-51: run_sipp_tests.sh — sipp使用-timeout 10s，慢速网络可能不够

**文件**: `tests/scripts/run_sipp_tests.sh`

**问题**: 第22行sipp命令使用`-timeout 10s`，对于慢速网络或多步骤场景可能不够。

**分析**: 
- `-timeout 10s`: 10秒超时
- 对于需要多轮交互的场景（如SM3认证需要4-5条消息），10秒在慢速网络下可能不够
- CI环境中网络延迟可能导致超时

**影响**: 
- 慢速环境下测试可能超时失败
- CI环境不稳定

**修复建议**: 增加超时时间到30s或根据场景类型动态调整。

## 三、P3-轻微问题（新增）

### P3-31: run_sipp_tests.sh — FAILED_TESTS使用\n依赖echo -e

**文件**: `tests/scripts/run_sipp_tests.sh`

**问题**: 第28行`FAILED_TESTS`使用字面量`\n`，第36行使用`echo -e`解释换行，依赖shell支持`-e`选项。

**代码位置**: 第28行、第36行
```bash
FAILED_TESTS="$FAILED_TESTS\n  - $name"  # 字面量\n
echo -e "  失败用例:$FAILED_TESTS"        # 依赖echo -e
```

**影响**: 
- 某些shell（如dash）的`echo`不支持`-e`，输出将包含字面量`\n`
- 可移植性差

**修复建议**: 使用`printf`替代`echo -e`，或使用实际换行符。
