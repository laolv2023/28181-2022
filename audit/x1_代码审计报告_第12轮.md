## 二、P2-中等问题（新增）

### P2-61: SipCharsetHelper.java — encode/decode使用字符串字符集名而非Charset对象

**文件**: `news/main/java/com/genersoft/iot/vmp/gb28181/utils/SipCharsetHelper.java`

**问题**: `encode(String)`和`decode(byte[])`方法使用`DEFAULT_CHARSET`（字符串名称）而非`DEFAULT_CHARSET_OBJ`（Charset对象），导致catch块成为死代码。

**代码位置**: 第121行、第145行

```java
// 第64行: 已有Charset对象
private static final Charset DEFAULT_CHARSET_OBJ;

// 第121行: 但encode使用字符串名称
return text.getBytes(DEFAULT_CHARSET);  // 应使用 DEFAULT_CHARSET_OBJ

// 第145行: decode也使用字符串名称
return new String(bytes, DEFAULT_CHARSET);  // 应使用 DEFAULT_CHARSET_OBJ
```

**分析**: 
- `getBytes(String charsetName)` 声明抛出 `UnsupportedEncodingException`
- `getBytes(Charset charset)` 不抛出受检异常
- 静态块已验证`DEFAULT_CHARSET_OBJ`可用，因此`getBytes(DEFAULT_CHARSET)`的`UnsupportedEncodingException`永远不会抛出
- catch块（第122-126行、第146-149行）是死代码

**影响**: 
- 性能：每次调用都进行字符集名称查找
- 代码质量：死代码catch块误导维护者
- 正确性：虽然功能正确，但API使用不当

**修复建议**: 
```java
// encode方法
return text.getBytes(DEFAULT_CHARSET_OBJ);  // 使用Charset对象，无需try-catch

// decode方法
return new String(bytes, DEFAULT_CHARSET_OBJ);  // 使用Charset对象，无需try-catch
```

## 三、P3-轻微问题（新增）

### P3-35: SipCharsetHelper.java — catch块注释承认"理论上不会发生"

**文件**: `news/main/java/com/genersoft/iot/vmp/gb28181/utils/SipCharsetHelper.java`

**问题**: 第123行注释`// 理论上不会发生，DEFAULT_CHARSET_OBJ 已经在静态块兜底`承认catch块是死代码，但未移除。

**分析**: 
- 注释明确承认异常不会发生
- 但仍保留了try-catch结构
- 应移除死代码或改用Charset对象

**修复建议**: 移除try-catch，直接使用`DEFAULT_CHARSET_OBJ`。
