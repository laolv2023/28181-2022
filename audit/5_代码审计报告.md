### P0-02: 文件上传路径遍历漏洞

**级别**: P0-致命（安全漏洞）
**分类**: B04 路径遍历

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/transmit/cmd/impl/SIPCommander2022Supplement.java`
**行号**: 221

**问题代码**:
```java
Path firmwarePath = Paths.get(FIRMWARE_UPLOAD_DIR, deviceId);
```

**问题分析**:

`deviceId` 直接来自用户请求参数（`ApiDeviceControlController.deviceUpgrade` 方法的 `@RequestParam String deviceId`），未经任何校验就用于构造文件路径。如果 `deviceId` 包含 `../`，攻击者可以将固件文件写到任意目录。

**攻击示例**:
```
POST /api/device/control/upgrade
deviceId=../../../etc/cron.d/evil
```
这将导致文件被写入 `/tmp/wvp/firmware/../../../etc/cron.d/evil`，即 `/etc/cron.d/evil`，可能实现任意命令执行。

**影响**: 攻击者可利用此漏洞向系统任意目录写入文件，可能导致远程代码执行。

**修复建议**:
1. 对 `deviceId` 进行格式校验（仅允许字母数字，长度限制）
2. 使用 `Path.normalize()` 后检查是否仍在 `FIRMWARE_UPLOAD_DIR` 目录内
3. 拒绝包含 `..` 或 `/` 的 `deviceId`

### P0-03: 文件上传无类型校验

**级别**: P0-致命（安全漏洞）
**分类**: B25 文件上传安全

**文件**: `backend/src/main/java/com/genersoft/iot/vmp/gb28181/transmit/cmd/impl/SIPCommander2022Supplement.java`
**行号**: 219-230

**问题代码**:
```java
public String uploadFirmware(String deviceId, MultipartFile file) {
    // ...
    String originalFilename = file.getOriginalFilename();
    String savedName = deviceId + "_" + System.currentTimeMillis() + 
                       originalFilename.substring(originalFilename.lastIndexOf("."));
    // 直接保存，无文件类型校验
    Path firmwarePath = Paths.get(FIRMWARE_UPLOAD_DIR, deviceId);
    Files.createDirectories(firmwarePath);
    Files.copy(file.getInputStream(), firmwarePath.resolve(savedName));
```

**问题分析**:

### P2-02: TcpReconnectHelper close() 异常导致误报连接失败

**级别**: P2-一般（健壮性）
**分类**: D 可靠性与健壮性

**文件**: `news/main/java/com/genersoft/iot/vmp/gb28181/utils/TcpReconnectHelper.java`
**行号**: 142-145

**问题代码**:
```java
try {
    socket.connect(new InetSocketAddress(ip, port), connectTimeout);
    socket.close();  // 第142行：close()可能抛IOException
    return true;
} catch (IOException e) {  // 第144行：会捕获close()的异常
    return false;  // 误报连接失败
}
```

**问题分析**:

`socket.close()` 在 `try` 块内，如果 `connect()` 成功但 `close()` 抛出 `IOException`，异常会被 `catch` 捕获并返回 `false`，导致成功的连接被误报为失败。

**影响**: TCP连接检测可能产生假阴性，导致本可用的连接被判定为不可用。

**修复建议**: 将 `socket.close()` 移到 `finally` 块中，或在 `connect()` 成功后立即 `return true`，在 `finally` 中关闭socket。

## 七、系统性问题总结

### 7.1 编译错误（P0-01）

5个Handler文件存在相同的编译错误模式（`cmdType` vs `CMD_TYPE`），而2个Notify Handler正确使用了 `CMD_TYPE`。这表明可能是复制粘贴时未统一修改，或代码生成工具的模板不一致。

### 7.2 集成缺失（P1-02）

12个工具类已完整实现但从未被调用，占38项改造的32%。这是最严重的系统性问题——代码审查中看似已实现的功能，在实际运行中不会生效。建议在发布前逐一验证各改造项的端到端功能。

### 7.3 文件上传安全（P0-02/03, P1-03/04/05）

`SIPCommander2022Supplement.uploadFirmware` 方法存在多个安全问题：路径遍历、无类型校验、无大小限制、硬编码路径、无资源映射。前端虽有校验但后端缺失，可被绕过。

### 7.4 合规性Bug（P1-01）

`FileURL` vs `FileuRL` 元素名不一致是典型的规范合规性问题。代码注释自相矛盾（写 `FileURL` 但说"u小写"），表明开发时已意识到正确名称但实现时出错。这将直接导致测试用例 A4.4-02 失败。
